# KB AI 엘리트 1~5주차 학습 내용 정리

LangChain v1.0 / LangGraph 기반 과정. 전체 흐름: **LLM 기초(1주차) → LangGraph 오케스트레이션(2주차) → RAG(3주차) → MCP(4주차) → 도구 선택·조합 심화 + A2A(5주차)**.

공통 스택: `init_chat_model("gpt-4o-mini")`, OpenAI 임베딩 `text-embedding-3-small`, KB은행/금융 도메인 예제(여신 심사, 이체, 공시 PDF, 은행 DB).

---

## 1주차 — LangChain 기초

### 01. 환경 설정과 기초
- **주제**: LangChain 1.0+ 환경 구성, 모델 초기화·호출·메시지 구성. OpenAI SDK 직접 호출과 비교하며 "왜 LangChain인가"를 체감.
- **핵심 개념**
  - `init_chat_model("gpt-4o-mini")` — 프로바이더 자동 팩토리 (대안: `ChatOpenAI` 직접 초기화)
  - Runnable 3대 호출: `invoke()`(단일) / `stream()`(실시간) / `batch()`(병렬 리스트)
  - `response.usage_metadata`로 토큰/비용 추적, `temperature`(0=일관성, 0.9=창의성)·`max_tokens` 제어
  - 메시지 타입: `SystemMessage` / `HumanMessage` / `AIMessage`, 멀티턴 대화 구성
- **실습**: 은행 상담사/IT 헬프데스크 역할 부여 비교, 은행 상품 질문 batch 처리, temperature별 슬로건 생성 비교, 수신/여신 멀티턴 상담

```python
model = init_chat_model("gpt-4o-mini", temperature=0.0, max_tokens=500)
response = model.invoke("질문")
for chunk in model.stream("질문"): ...
responses = model.batch(questions)
```

### 02. 프롬프트와 체인 (LCEL)
- **주제**: 재사용 가능한 프롬프트 설계, LCEL 파이프 체인, Pydantic 기반 Structured Output.
- **핵심 개념**
  - `ChatPromptTemplate.from_messages([("system", ...), ("human", "...{변수}...")])` — 변수만 바꿔 재사용
  - LCEL 체인: `prompt | model | StrOutputParser()` (입력 dict → 메시지 → AIMessage → str)
  - `RunnablePassthrough.assign()`(추가 필드 주입), `RunnableLambda`(커스텀 전처리 삽입)
  - **Structured Output**: Pydantic `BaseModel` + `Field` + `Literal` 스키마 → `model.with_structured_output(스키마)` → 객체 속성으로 접근
- **실습**: 은행 상품 설명/여신 심사 의견서 템플릿, 금융 용어 해설 체인, `CustomerInquiry`(문의 분류)·`LoanAssessment`(여신 심사)·`CustomerSentiment`(VOC 감성분석) 구조화 출력

```python
chain = prompt | model | StrOutputParser()
chain.invoke({"term": "DSR", "audience": "대출 고객"})

class CustomerInquiry(BaseModel):
    category: Literal["여신", "수신", "카드"] = Field(description="문의 카테고리")
result = model.with_structured_output(CustomerInquiry).invoke("...")
```

### 03. 도구와 에이전트
- **주제**: `@tool`로 외부 기능 정의, `create_agent`로 자율형 Agent 구축, 대화 메모리 연결.
- **핵심 개념**
  - `@tool` 데코레이터 — **docstring이 LLM의 도구 선택 기준** (5주차까지 관통하는 원리)
  - `create_agent(model=..., tools=..., system_prompt=...)` (`from langchain.agents`)
  - Agent 메시지 흐름: `HumanMessage → AIMessage(tool_calls) → ToolMessage → AIMessage(최종)`
  - 대화 메모리: `InMemorySaver` checkpointer + `config={"configurable": {"thread_id": ...}}`로 세션 구분
- **실습**: 은행 Tool 5종(잔고·대출이자·환율·금리·신용등급), 복합 질문에서 자동 Tool 조합, thread_id 기반 멀티턴 기억 확인

```python
@tool
def get_account_balance(account_number: str) -> str:
    """계좌 잔고를 조회합니다."""
    ...
agent = create_agent(model=model, tools=banking_tools, checkpointer=InMemorySaver())
agent.invoke({"messages": [...]}, {"configurable": {"thread_id": "customer_001"}})
```

### 04. 미들웨어와 가드레일
- **주제**: Middleware로 Agent 동작 확장, Guardrails로 입출력 방어막 구축. (모두 `from langchain.agents.middleware`)
- **핵심 개념**
  - `LLMToolEmulator` — 실제 Tool 없이 결과 시뮬레이션(프로토타이핑)
  - `PIIMiddleware` — 정규식 기반 민감정보(전화번호/주민번호/계좌번호) 마스킹
  - `SummarizationMiddleware` — 긴 대화 자동 요약으로 컨텍스트 관리
  - 커스텀 훅: `@before_model`/`@after_model`(감사 로그), `@wrap_model_call`(동적 모델 라우팅, `request.override(...)`)
  - 가드레일: `@before_agent(can_jump_to=["end"])` 입력 필터(금지어 차단, LLM 호출 전 종료) + `@after_agent` 출력 컴플라이언스 검증
- **실습**: PII 마스킹 Agent, PB 상담 요약 비교, 감사 로그, 금지어 차단 + 투자권유 검증 3중 방어막 통합 Agent

```python
@before_agent(can_jump_to=["end"])
def input_guard(state, runtime):
    if any(kw in state["messages"][-1].content for kw in 금지어):
        return {"messages": [{"role": "assistant", "content": "처리 불가"}], "jump_to": "end"}
```

### 05. 메모리와 고급 기능
- **주제**: Runtime Context, 장기 메모리(`InMemoryStore`), Structured Output Agent, 오류 처리.
- **핵심 개념**
  - Runtime Context: `@dataclass` 스키마 → `agent.invoke(..., context=...)` → 미들웨어에서 `request.runtime.context` 접근 (고객 등급별 차등 응대)
  - **장기 메모리** `InMemoryStore` (`from langgraph.store.memory`): `put(namespace, key, value)` / `get` / `search` / `delete` — 세션 초월 영구 기억, namespace로 격리
  - `@wrap_model_call`에서 store를 읽어 system_prompt에 기억 주입, 또는 Tool로 Agent가 직접 저장/조회
  - `create_agent(..., response_format=스키마)` — 최종 응답을 Pydantic 구조체로
  - `StructuredOutputValidationError` 처리: try/except 폴백 패턴
- **실습**: VIP 차등 응대 Agent, 장기 기억 PB Agent, **종합 실습** — 여신심사 보조 Agent(Tool+가드+`LoanReview` 구조화), IT 헬프데스크 티켓 분류 Agent

```python
store = InMemoryStore()
store.put(("C001", "consulting"), "pref_001", {"facts": ["안정적 투자 선호"]})
agent = create_agent(model=model, tools=[...], response_format=LoanReview)
```

## 2주차 — LangGraph

### 01. 상태·노드·그래프 기초
- **주제**: State가 Node 사이를 흐르는 원리, LangGraph 기본 3요소.
- **핵심 개념**
  - State(작업지시서) / Node(작업자) / Edge(컨베이어) / Graph(공장 라인) — **모든 노드는 같은 State를 공유**
  - 그래프 5단계: `TypedDict` State 정의 → Node 함수("바꿀 칸만" dict 반환) → `StateGraph` 조립(`add_node`/`add_edge`, `START`/`END`) → `compile()` → `invoke()`
  - `stream(stream_mode="updates")`(변경분) vs `"values"`(전체 state)
  - Reducer: 기본 덮어쓰기, `Annotated[list, operator.add]`로 누적. `add_conditional_edges`로 rule 분기
- **실습**: 인사말/쇼핑 주문/이메일 파이프라인 그래프, 은행 출금 처리 + 로그 누적

```python
class GreetingState(TypedDict):
    name: str
    greeting: str

builder = StateGraph(GreetingState)
builder.add_node("make_greeting", make_greeting)
builder.add_edge(START, "make_greeting")
builder.add_edge("make_greeting", END)
app = builder.compile()
```

### 02. 에이전트 기획 — State 설계 & 분기 설계
- **핵심 개념**
  - State 정의 3방식: `TypedDict`(기본) / `dataclass`(기본값) / Pydantic `BaseModel`(`Field(ge=0, le=10)` 자동 검증)
  - 커스텀 Reducer 3패턴: dict 병합, 최댓값, 중복 제거 누적 — `(현재값, 새값) -> 합친값` 함수
  - LLM 라우터 입문: `with_structured_output`으로 구조화된 판단을 받아 분기
- **실습**: 은행 문의 처리 설계, 고객 프로필 병합, 대출 위험도 평가(최댓값 reducer)

### 03. 에이전트 구현 — 3가지 라우팅 버전
- **핵심 개념**
  - `model.bind_tools(tools)`, prebuilt `ToolNode`(도구 실행) + `tools_condition`(tool_calls 있으면 `"tools"`, 없으면 `END`)
  - `Annotated[list, add_messages]`로 대화 히스토리 누적
  - **V1** 키워드 rule 라우팅 / **V2** LLM+Tools 라우팅 / **V3** 하이브리드
- **실습**: 고객 문의 처리 Agent 3버전 비교

```python
builder.add_node("llm", llm_node)
builder.add_node("tools", ToolNode(tools))
builder.add_conditional_edges("llm", tools_condition)
builder.add_edge("tools", "llm")
```

### 04. 워크플로 패턴 (Chaining / Parallel / Routing)
- **핵심 개념**
  - **Prompt Chaining**: 분류 → 답변 → 이메일 초안 순차 처리
  - **Parallelization** (fan-out/fan-in): `add_edge` 여러 개로 병렬 분기 후 취합 — 같은 칸을 쓰면 `operator.add` reducer 필수
  - **Routing**: `with_structured_output` + `Literal`로 복잡도 판정 → 모델 선택(간단→mini, 복잡→4o) 비용 최적화
- **실습**: 금융 보고서 자동 생성, 금융 상품 다각도 병렬 분석, 복잡도별 모델 선택

### 05-1. Evaluator-Optimizer
- **핵심 개념**: Generator(초안) → Evaluator(`EvaluationResult`: passed/score/feedback 구조화 평가) → 미통과 시 피드백 들고 Generator로 루프, 합격 시 `END`
- **실습**: 광고 문구 생성-평가-개선 자동 반복

```python
def route_eval(state):
    return END if state["passed"] else "generator"
builder.add_conditional_edges("evaluator", route_eval)
```

### 05-2. Orchestrator-Worker
- **핵심 개념**
  - Planner LLM이 `with_structured_output(Plan)`으로 섹션 목록 생성
  - **Send API** (`from langgraph.types import Send`): 섹션마다 `Send("worker", {...})` 반환 → 워커 **동적** 병렬 생성
  - `operator.add` reducer로 워커 결과 자동 취합
- **실습**: 금융 보고서 자동 작성 (계획 → 워커 병렬 집필 → 취합 → 합성)

```python
def assign_workers(state):
    return [Send("worker", {"section": s}) for s in state["plan"].sections]
```

### 06. 메모리 — Checkpointer / Time Travel / Interrupt / Store
- **핵심 개념**
  - **Checkpointer** `InMemorySaver`: `compile(checkpointer=...)` + `thread_id`로 세션별 State 영속·격리
  - `get_state(config)` / `get_state_history(config)`, **Time Travel**(과거 스냅샷에서 재분기)
  - 정적 Interrupt: `compile(interrupt_before=[...])`
  - **Store**(장기 메모리): `InMemoryStore`, thread 무관 cross-session 저장, CRUD + `search(query=...)` 유사 검색. `compile(checkpointer=..., store=...)` 조합, 노드에서 `store: BaseStore` 주입
- **실습**: 고객별 세션 격리 챗봇, Time Travel 수정, Store 기반 개인화 Agent

### 07. Human-in-the-Loop — 동적 interrupt / Command
- **핵심 개념**
  - **동적 `interrupt()`** (`from langgraph.types import interrupt, Command`): 노드/Tool 내부에서 조건부 일시정지 (예: 100만원 이상 이체만 승인 요청)
  - 재개: `app.invoke(Command(resume=값), config)` — interrupt()가 그 값을 반환
  - `Command(goto="node")`: 승인/거절에 따라 다른 노드로 점프, 자연어 의도 파싱과 결합
- **실습**: 고액 이체 승인 시스템, 입력 검증 루프, 추천-승인-예약 흐름

```python
def approval_node(state):
    if state["amount"] >= 1_000_000:
        decision = interrupt({"message": "승인?", "amount": state["amount"]})
        if decision == "no":
            return Command(goto="reject")
```

### 08. 멀티 에이전트 — Subgraph / Supervisor / Network
- **핵심 개념**
  - **Subgraph**: 컴파일된 그래프를 다른 그래프의 노드로 등록 — 캡슐화, 단독 실행 가능
  - **Supervisor**: 중앙 LLM이 `RouteResponse(next: Literal["researcher","writer","reviewer","FINISH"])`로 워커 배정, FINISH까지 왕복
  - **Network/Handoff**: `@tool`이 `Command(goto=...)`를 반환해 다른 Agent로 대화 이관 — 중앙집중(Supervisor) vs 분산(Handoff) 비교
- **실습**: 레스토랑 주문(Subgraph), 금융 리서치 팀(Supervisor), KB은행 멀티에이전트 고객센터(Handoff)

## 3주차 — RAG

데이터: **KB국민은행 제25기 영업보고서 / 임원 책무변경 공시 PDF**. 임베딩은 `text-embedding-3-small`(1536차원) 고정.

### 01. RAG 한눈에 체험
- **주제**: **파싱 → 청킹 → 임베딩 → 리트리버 → 생성** 5단계를 처음부터 끝까지 한 번 체험. LLM의 두 한계(지식 단절, 환각) 시연.
- **핵심 개념**: 인덱싱(한 번) vs 질의(매번), `pypdf.PdfReader`, `RecursiveCharacterTextSplitter(chunk_size=400, chunk_overlap=80)`, `OpenAIEmbeddings` + 코사인 유사도 직접 계산, `InMemoryVectorStore.from_texts` + `similarity_search(k=3)`, "자료에 근거해서만 답하라" 프롬프트
- **실습**: `ask_rag(question, k=3)` 함수 — LLM 단독 vs RAG 답변 비교(당기순이익을 RAG만 정확히 답함)

```python
def ask_rag(question, k=3):
    docs = vector_store.similarity_search(question, k=k)
    context = "\n\n".join(d.page_content for d in docs)
    return model.invoke([SystemMessage("...[자료]에 근거해서만..."),
                         HumanMessage(f"[자료]\n{context}\n\n[질문]\n{question}")]).content
```

### 02. 파싱·청킹·임베딩 고도화
- **주제**: "문서 종류와 사용 목적에 따라 알맞은 도구가 달라진다" (Garbage In, Garbage Out).
- **핵심 개념**
  - 파싱: 텍스트 → `pypdf` vs `PyMuPDF(fitz)` 속도 비교 / 표 → `pdfplumber.extract_tables()`, `pymupdf4llm.to_markdown()` / 이미지 → PyMuPDF 추출 후 **gpt-4o 멀티모달 캡셔닝**(그림 속 수치를 텍스트화해 검색 가능하게)
  - 청킹 4전략: 고정 크기 / 재귀적(기본 추천) / 문장 단위 / 의미 기반(`SemanticChunker`)
  - 임베딩: small(1536) vs large(3072), `dimensions` 축소, 인덱싱·질의에 **같은 모델** 사용 원칙
- **실습**: KB 영업보고서 표 → Markdown 변환, 이미지 캡셔닝, 유사도 확인

### 03. 리트리버 고도화 + 출처 표기
- **핵심 개념**
  - `PyPDFLoader`로 페이지 단위 로딩 → `Document.metadata`(page/source) 유지, `split_documents()`(메타데이터 보존)
  - `as_retriever(search_kwargs={"k": 5})`, **MMR**(`max_marginal_relevance_search` — 중복↓ 다양성↑)
  - **2단계 검색(리랭킹)**: 임베딩으로 넓게(k=20) → LLM 리랭킹으로 좁히기
  - 출처 표기: 조각마다 `[출처: 파일명 p.쪽]` 부착, 답변 끝에 출처 명기
- **실습**: `ask_with_sources(question)` — 넓게 검색 → 리랭킹 → 출처 포함 생성

### 04. LangGraph 기반 RAG 파이프라인
- **주제**: RAG 구축 과정을 LangGraph Chaining으로 자동화 — 첫 노드가 의도·문서 특징을 보고 **전략을 세우면** 이후 노드가 따름.
- **핵심 개념**
  - `plan`(전략 수립, `with_structured_output(Strategy)`) → `parse`(pypdf/pdfplumber 분기) → `chunk`(recursive/semantic 분기) → `embed`(small/large 분기) 4노드 그래프
  - `NotRequired` State 필드, `Annotated[list, operator.add]` 로그 누적
  - RAG 구축은 순서가 고정이므로 Supervisor 대신 Chaining으로 충분하다는 설계 판단
- **실습**: 정확성 우선(표 정밀) vs 속도·비용 우선(텍스트) 시나리오에서 다른 파이프라인이 자동 구성됨을 확인

```python
class Strategy(BaseModel):
    parser: Literal["pypdf", "pdfplumber"] = Field(description="표 중요하면 pdfplumber")
    chunking: Literal["recursive", "semantic"]
    embedding: Literal["small", "large"]
    reason: str
```

### 05. Langfuse Observability
- **핵심 개념**
  - Trace(요청 1건) / Span(작업 구간) / Generation(LLM 호출, 토큰·비용) 3단위
  - `get_client()` + `auth_check()`, `from langfuse.langchain import CallbackHandler`
  - 핵심 한 줄: `invoke(..., config={"callbacks": [langfuse_handler]})` → 하위 단계 자동 기록, `langfuse.flush()`
  - 메타데이터: `langfuse_user_id` / `langfuse_session_id` / `langfuse_tags`로 필터링
- **실습**: 단발 호출 → 구축 그래프 → 질의 RAG 그래프(retrieve/generate Span) 추적, 대시보드에서 근거·지연·토큰·비용 확인

### (보충) unstructured.io PDF 파싱
- **핵심 개념**: `partition_pdf(filename, strategy=...)` 하나로 다양한 PDF 대응 — `"fast"`(텍스트) / `"hi_res"`(딥러닝 레이아웃, 표·그림) / `"ocr_only"`(스캔 문서, 한국어 OCR) / `"auto"`. `infer_table_structure=True`(표→HTML), `extract_image_block_to_payload=True`(이미지→base64), `chunk_by_title`로 RAG용 청킹
- **실습**: 표·그림 문서 / 2단 논문 / 스캔 문서를 각각 알맞은 strategy로 파싱

## 4주차 — MCP

공통 축: `FastMCP` + `@mcp.tool()` + `transport="streamable-http"` 서버, `subprocess` 백그라운드 기동, `MultiServerMCPClient`/`load_mcp_tools`로 LangChain 도구 변환, "목록 확인 → 스키마 확인 → 호출" 습관.

### nb01. MCP 한눈에 이해하기
- **핵심 개념**: 도구 없는 LLM의 한계 → Tool Calling(`bind_tools`, `tool_calls` — LLM은 "무엇을 부를지"까지만 결정) → **N×M 통합 문제를 MCP 표준으로 N+M으로** (USB-C 비유). MCP 3레이어(Tool/Resource/Prompt), 동작 2단계(`tools/list` / `tools/call`), LangChain 도구와의 관계(`langchain-mcp-adapters`로 변환)

### nb02. 공개 MCP 서버 연결·도구 탐색
- **핵심 개념**: 공개 서버 생태계(Smithery, mcp.so), `mcpServers` JSON 설정, **FMP(Financial Modeling Prep) 공개 서버 연결**(`FMP_API_KEY`), 도구 스키마 읽기, `MultiServerMCPClient`로 다중 서버 연결, 인증 토큰은 Secrets로 관리, 디버깅 체크리스트 + MCP Inspector
- **실습**: FMP로 주가/외환/뉴스 조회, 로컬 대체 서버 `kb_info_server.py`(port 8001)

### nb03. MCP 클라이언트 직접 설계
- **핵심 개념**: 클라이언트 4박자 — **연결(`streamablehttp_client`) → 초기화(`session.initialize()` 핸드셰이크) → 목록(`list_tools()`) → 호출(`call_tool()`)**. `list_resources()`/`list_prompts()`, `AsyncExitStack` 기반 재사용 클라이언트 클래스, 무상태(`get_tools()`) vs 세션 유지(`client.session()` + `load_mcp_tools`) 비교

```python
async with streamablehttp_client(URL) as (read, write, _):
    async with ClientSession(read, write) as session:
        await session.initialize()
        tools = await session.list_tools()
```

### nb04. 로컬 MCP 서버 직접 만들기
- **핵심 개념**: 로직을 평범한 함수로 검증 → `@mcp.tool()`만 붙여 도구화. **docstring = 공식 사용 설명서**. STDIO(로컬 앱, Claude Desktop 등록) vs Streamable-HTTP(네트워크) 비교, `@mcp.resource()` 리소스 등록, MCP Inspector 점검, 보안(위험 작업엔 사람 확인), 실제 외부 API(환율·뉴스 RSS) 연동 + 실패 시 폴백
- **실습**: `kb_rule_server.py`(규정 검색), `kb_resource_server.py`(리소스), `kb_live_server.py`(실시간 환율·뉴스)

### nb05. MCP + LangGraph ReAct Agent
- **핵심 개념**: `create_agent` 한 줄 vs LangGraph 직접 조립(`MessagesState`, ToolNode, `tools_condition` 순환) 둘 다 실습. `MemorySaver` + thread_id 메모리, `stream_mode="updates"` 중간 과정 관찰, **`interrupt_before=["tools"]` 승인 게이트**, 커스텀 State·Reducer
- **실습**: `kb_agent_server.py`(환율·잔고·규정, port 8004)를 물린 에이전트, 승인 게이트 그래프

### nb06. 3주차 RAG를 MCP 도구로 연결
- **핵심 개념**: 3주차 검색기를 `@mcp.tool()` `search_documents(query)`로 감싸 표준 도구화. 청크를 `kb_chunks.json`으로 저장 → 서버 기동 시 1회 임베딩(프로세스 분리). "직접 RAG" vs "MCP RAG" 비교 — 재사용·표준화·역할 분리(검색 서버 vs 판단 에이전트)
- **실습**: `kb_rag_server.py`(port 8016) + 검색 도구를 물린 ReAct 에이전트의 근거 기반 답변

```python
@mcp.tool()
def search_documents(query: str) -> str:
    """사내 문서(KB 공시)에서 질문과 관련된 내용을 검색합니다."""
    hits = store.similarity_search(query, k=3)
    return "\n".join(f"[근거 {i+1}] {d.page_content}" for i, d in enumerate(hits))
```

### nb07. 업무 시나리오 + Langfuse 모니터링
- **핵심 개념**: "보고서 초안 도우미" — 문서 검색 → 지표 조회 → 초안 작성 3단계 업무 자동화. Langfuse 콜백으로 도구 호출 수·토큰·비용·시간 관찰. 현업 고려 3가지: 보안 / 권한(위험 작업 사람 확인) / 비용
- **실습**: `kb_report_server.py`(search_document·get_metric·draft_report, port 8005) + ReAct 에이전트로 여신 동향 보고서 초안 생성, Langfuse 트레이스 기록

## 5주차 — 도구 선택·조합 심화 + A2A

공통 뼈대: FastMCP(streamable-http) + `MultiServerMCPClient` + `create_agent`. 관통 원리: **"도구 설명(description)이 선택/라우팅 품질을 좌우한다."**

### nb01a. LLM의 도구 선택 기준 분석 — 기초
- **핵심 개념**
  - LLM이 보는 것은 도구의 **이름·설명·입력 형식** 세 가지, 그중 **설명이 최대 신호**
  - `bind_tools` 후 `tool_calls`로 "실행 전 선택 결정"만 관찰
  - ReAct 그래프에 **선택 분석 노드** 추가 (`selections: Annotated[list, operator.add]` 누적)
  - **절제 실험(ablation)**: `t.model_copy(update={"description": ""})`로 설명 제거 시 선택 변화 관찰
  - 애매하게 겹치는 도구(`search_rule` vs `search_faq`), 환각 호출 점검(맞는 도구 없으면 미호출)
- **실습**: `kb_analysis_server.py`(port 8009) — 일부러 설명이 겹치는 도구 5개, 선택 분석 그래프

### nb01b. 도구 선택 분석 — 심화
- **핵심 개념**
  - **표현 안정성**: 같은 의도의 패러프레이즈들에서 선택이 흔들리는지 측정
  - **설명 A/B 최적화**: 설명에 대상(임직원/고객)을 명시한 개선판 vs 원본 정확도 비교
  - 멀티스텝 복합 질문, 인자 부족 시 거동(기본값/되물음/미호출)
  - 질문↔정답 도구 라벨 세트로 **선택 정확도 회귀 테스트**, Langfuse 연동
  - 좋은 도구 설명 체크리스트: 무엇을 / 대상·범위 / 입력 예시 / 경계 긋기

### nb02a. DB 설계·접근 MCP — 기초
- **핵심 개념**
  - SQLite 은행 DB 4테이블(`customers`/`accounts`/`transactions`/`loans`, `customer_id` 조인 키)
  - DB 도구 설계 원칙: **읽기 전용(read-only)**
  - **구체 도구**(`get_customer`, `list_accounts`, `list_loans`) vs **만능 조회 도구**(`run_readonly_query` — SELECT만 허용, 변경 구문 거부) 비교
- **실습**: `kb_bank.db` + `kb_db_server.py`(port 8010), 자연어 → DB 질의 에이전트

```python
@mcp.tool()
def run_readonly_query(sql: str) -> str:
    """읽기 전용 SELECT 질의를 실행합니다."""
    s = sql.strip().lower()
    if not s.startswith("select"):
        return "거부: 읽기 전용(SELECT) 질의만 허용됩니다."
    ...
```

### nb02b. DB MCP — 심화
- **핵심 개념**
  - **스키마 힌트 노출**: 도구 설명에 테이블 스키마를 넣으면 SQL 성공률↑ (토큰 비용과 트레이드오프)
  - **`ToolCallLimitMiddleware`**(`run_limit=2, exit_behavior="continue"`)로 특정 도구 호출 횟수 제한
  - 파라미터 검증(없는 고객번호 → 사람이 읽을 메시지), 집계 질의(GROUP BY/SUM), 구조화 로그(JSON trace)
  - 검증 개선: 정규식 단어 경계 `\b(insert|update|...)\b`로 `updated_at` 오탐 방지

### nb02c. RAG 도구 vs DB 도구 — 에이전트 라우팅
- **핵심 개념**
  - DB(정형/딱 떨어지는 값) vs RAG(비정형/문서 근거) — 에이전트는 **도구 설명**으로 가른다
  - `MultiServerMCPClient`에 **두 서버 동시 등록**: RAG 서버(port 8020, `search_report`) + DB 서버(port 8021)
  - 라우팅 A(DB형)/B(RAG형)/C(혼합=둘 다 호출), 질문↔기대 소스 라벨로 **라우팅 정확도 배치 측정**
- **실습**: `kb_report_server.py` + `kb_db2_server.py` 이중 등록 라우팅 에이전트

```python
client = MultiServerMCPClient({
    "policy": {"transport": "streamable_http", "url": "http://127.0.0.1:8020/mcp"},
    "bank":   {"transport": "streamable_http", "url": "http://127.0.0.1:8021/mcp"},
})
```

### nb03a. A2A 프로토콜 — 기초
- **핵심 개념**
  - **A2A 표준 공문 4종** (`@dataclass`): `AgentCard`(명함) / `Task`(일감) / `Message`(봉투) / `Artifact`(회신 결과물)
  - **MCP vs A2A**: MCP = 도구를 빌려 쓰기, A2A = 동료 에이전트에게 일을 맡기기. 실무에선 Orchestrator가 A2A로 위임하고 **Worker 내부에서 MCP 도구 호출** (두 층 결합)
  - Orchestrator: 명함의 `skill`로 레지스트리를 만들어 라우팅(디스커버리), `delegate()` 위임 + 통신 로그, Artifact들을 LLM으로 종합. 새 Worker는 명함만 등록하면 추가(느슨한 결합)
  - 미등록 skill 안전 거절, `nest_asyncio` 동기 브리지로 Worker의 MCP 호출
- **실습**: Worker 4종(신용분석·시장금리·규정검토·MCP 데이터조회) + Orchestrator로 "여신 심사 요약서 초안" 생성, `kb_a2a_server.py`(port 8011)

```python
class Orchestrator:
    def __init__(self, workers):
        self.registry = {skill: w for w in workers for skill in w.card.skills}
    def delegate(self, skill, content):
        worker = self.registry[skill]
        task = Task(task_id=..., skill=skill, content=content)
        return worker.handle(Message(sender="Orchestrator", receiver=worker.card.name, task=task))
```
