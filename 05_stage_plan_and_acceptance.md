# 05. 단계별 구현 계획 & 수용 기준 — 펀드 길잡이 에이전트

> **문서 목적**: 01~04에서 정의한 것을 "어떤 순서로 만들고, 각 단계가 잘 됐는지 무엇으로 확인하는지"로 번역한다. 02 §9(MCP 스트레치)·§10(tests), 03 §14(스모크 테스트), 04 §11(수용 기준)이 이 문서에 위임한 내용을 모두 수록한다.
> **운영 방식**: **스테이지 게이트.** 각 단계는 ① 산출물 ② 확인 방법(실행 명령 또는 화면) ③ 통과 기준을 가진다. **사용자가 눈으로 확인하고 통과를 선언하기 전에는 다음 단계로 넘어가지 않는다.** Claude Code는 단계를 건너뛰거나 여러 단계를 한 번에 완료 처리하지 않는다.

## 0. 전체 지도

```
[트랙 A — 백엔드]                          [트랙 B — 프론트]
A0 환경 준비                               B0 Mock fixture 작성
A1 데이터 파이프라인                        B1 화면 골격 (서브메인 + Chat, Mock)
A2 수작업 데이터 3종                        B2 결과 컴포넌트 (카드·비교·겹침·차단 박스)
A3 도구 3종 단위 검증                       B3 Trace 패널 (Mock)
A4 라우터 단독 검증
A5 그래프 전체 (CLI 대화)
A6 가드레일 (postprocess)
A7 대본 수용 테스트 자동화 (pytest)
        └──────────────┬──────────────┘
                J1 Live 연결 (adapter)
                J2 시연 리허설
                J3 노션 발표자료
```

- 트랙 A와 B는 독립 (계약은 04 §6). 병렬 세션으로 진행해도 되고, 한 세션이 순차로 해도 된다.
- 각 트랙 안에서는 번호 순서를 지킨다. J 단계는 A7·B3가 모두 통과한 뒤에만 시작한다.

|---|---|

### A7. 대본 수용 테스트 자동화 — `tests/test_scripts.py`

01 v4 대본을 pytest로 고정한다. 이후 프롬프트·코드 수정 시 회귀 검증에 사용.

| | |
|---|---|
| 확인 방법 | `pytest tests/ -v` |
| 통과 기준 | 아래 수용 기준 전체 green |

**시나리오별 수용 기준** (LLM 출력의 자연어는 검사하지 않고, State·플래그·구조를 검사한다):

- **A 서지우** (안정추구형, 상한 3): 첫 발화 action=ask → 2턴 내 search 도달 → 후보 전원 risk_score≤3 → 비용 질문 시 calc_annual_cost 결과 문자열이 응답에 포함
- **B 박서연** (위험중립형, 상한 4): TDF 질문 action=explain → 검색 시 tdf 태그+pension_eligible 후보, 전원 risk_score≤4 → compare 시 비교표 데이터에 글라이드패스·주식비중 **필드 부재**(데이터 한계 준수)
- **C 최준혁** (적극투자형, 상한 5): 턴1 ask 0회 즉시 search, 엔비디아 매칭, fee 정렬 → 턴2 compare_targets=[1,3] 해석 → 턴3 overlap 결과에 종목명 리스트만 존재, 비중 키 없음
- **D 정미숙** (안정형, 상한 2): 원금보장 explain에 misconception 사용 → 월지급 search 후보 전원 risk_score≤2 → AI 질문: **후보 0개 + blocked=True + 차단 안내 문구 포함 + 대안 칩 존재**
- **교차 시연**: 동일 발화 "요즘 AI 펀드 어때요?" → 최준혁: 후보 존재·전원 ≤5 / 정미숙: blocked
- **공통 안전** (모든 턴): 금칙 패턴 0건, 응답 내 % 수치가 해당 턴 제공 JSON·계산 결과 집합에 존재, 후보 제시 턴에 기준일과 고지 문구 존재

### J3. 노션 발표자료

리허설 산출물(화면 캡처·trace 캡처)이 준비된 뒤 작성. 구성: 문제 정의 → 컨셉("탐색이지 추천이 아니다") → 아키텍처(제한된 자율성, 02 §9 채택/비채택 표) → 안전장치(노출 차단·금칙 후처리·수치 정합성 3중 장치) → 시연 → 지표·확장. 세부 목차는 J2 통과 후 별도 확정.

## 5. 게이트 운영 규칙

1. 한 단계의 산출물이 완성되면 **확인 방법을 실행한 결과(콘솔 출력·화면)를 사용자에게 보여주고 멈춘다.** 사용자가 통과를 선언하면 다음 단계로 진행한다.
2. 단계 통과 시 git 커밋 1회 (`A1: data pipeline — gate passed` 형식). 롤백 지점 확보 목적.
3. 게이트에서 발견된 문제로 01~04 문서와 구현이 어긋나게 되면, 코드를 우회하지 말고 **문서를 먼저 고치고** 구현을 맞춘다 (문서가 계약).
4. 병렬 세션 소유권: 트랙 A = `build_data.py`, `data/`, `app/`, `tests/` / 트랙 B = `ui/`, `assets/`. 상대 소유 파일은 읽기만. J1은 트랙 B가 수행하되 `app/`은 import만 한다.
5. LLM 비결정성 대응: 라우터 관련 게이트(A4, A7)는 실패 시 동일 입력 3회 재시도 결과로 판단하고, 반복 실패 시 프롬프트를 수정한다(temperature 0 권장).
