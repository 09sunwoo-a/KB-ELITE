# 03. RAG / 데이터 설계 — 펀드 길잡이 에이전트

> **문서 목적**: `펀드정보.xlsx`(311개 상품)를 검색·답변·계산에 쓸 수 있는 형태로 가공하는 파이프라인과, 응답 수치의 정합성을 보장하는 구조를 정의한다.
> **상위 문서**: 01 v4(11장 데이터셋 현황·한계, 8-3 고객 언어 사전), 02(6장 도구, 7장 가드레일).
> **설계 기준**: 수업 3주차 RAG 핵심만 채택. 311건 소규모 데이터에 맞는 최소 구조. 외부 벡터 DB·리랭킹·시맨틱 청킹 등은 쓰지 않는다(사유는 8장).

---

## 1. 핵심 원칙 — 검색은 고객 언어로, 수치는 정형 조회로

```
                ┌── contents.jsonl (고객언어형 CONTENT) ──→ 임베딩 → 벡터 검색 [검색 전용]
펀드정보.xlsx ──┤
 (원본 유일)     └── funds.json (정형 메타 + 파생 필드) ──→ 필터·정렬·비교·계산·응답 수치 [수치의 단일 출처]
```

- **funds.json이 모든 수치의 단일 출처(single source of truth)다.** 응답에 등장하는 총보수·수익률·위험등급·보유종목은 예외 없이 여기서 조회한 값이다.
- **CONTENT는 검색에만 쓴다.** LLM의 응답 생성 컨텍스트에는 CONTENT를 넣지 않는다. 검색용 문서와 답변용 데이터를 물리적으로 분리해, "검색이 잘 되게 쓴 문장"이 답변 수치를 오염시킬 경로 자체를 없앤다.
- LLM은 문장(선정 이유, 차이 서술, 설명)만 만든다. 수치가 필요한 자리는 코드가 채운다(6장).

## 2. 빌드 파이프라인 — `build_data.py` (1회 실행)

```
xlsx 로드(pandas) → 정제 → 파생 필드 생성 → funds.json / contents.jsonl 저장
                                          → CONTENT 임베딩 → embeddings 캐시 저장
                                          → 검증 assert (12장) → 실패 시 빌드 중단
```

앱 시작 시에는 산출물만 로드한다(임베딩 재계산 없음). 수업 4주차 nb06의 "청크 저장 → 기동 시 로드" 패턴을 빌드 타임까지 앞당긴 것으로, 시연 중 재시작을 빠르고 안정적으로 만든다.

## 3. 정제 규칙

| 항목 | 규칙 |
|---|---|
| 수익률 부동소수점 노이즈 | 원본에 `0.8100000000000001` 형태 존재 → 소수 2자리 반올림 |
| 결측 | 주요보유종목 77건 → `has_holdings_info: false` / 운용사 1건 → `null` (응답 시 생략) |
| 위험등급 | 텍스트 6단계 그대로 보존 + `risk_score`(1~6) 파생. 고객 노출은 텍스트만(01 4-2 주의사항) |
| 기준일 | 컬럼 없음 → 전 상품 `as_of: "2026-06-30"` 상수 부여(01 11-3 가정) |
| 전각 문자 | 상품명의 `＆` 등 전각 기호 → 반각 정규화(검색 매칭 안정성) |

## 4. 파생 필드 규칙

| 필드 | 생성 규칙 | 근거 시나리오 |
|---|---|---|
| `risk_score` | 매우낮은위험=1 … 매우높은위험=6 | 노출 규칙(01 4-2) |
| `tags.tdf` + `tdf_vintage` | 상품명 `TDF` + 정규식 `(20\d\d)` → 2015~2055 | B |
| `tags.pension_eligible` | 상품명에 `적격` 포함 | B (IRP 맥락) |
| `tags.hedge` | 상품명 `(H)` → "H", `(UH)`/`UH` → "UH", 없으면 null | B 환헤지 |
| `tags.income` | 상품명·특징에 `인컴|배당|월지급` | D |
| `tags.index_derivative` | 상품명에 `인덱스` 그리고 `파생` | 보유종목 해석 안내(01 11-3) |
| `share_class` | 상품명 꼬리 정규식 `(C-E|C-e|CE|Ce|C)[적]?$` 계열, 실패 시 null | 비용 안내 |
| `fee_quartile` | 총보수 사분위(1=저렴한 편 ~ 4) | CONTENT의 "수수료 낮은 편" 표현 근거 |
| `stocks_raw` | 주요보유종목 원문 문자열 보존 | 겹침·종목 검색 |

**종목 문자열은 분리하지 않는다.** 원본이 공백 구분인데 종목명 자체에 공백이 있어(`TENCENT HOLDINGS LTD Alibaba Group...`) 토큰 분리가 불안정하다. 대신 alias 사전(10장) 기반 **contains 매칭**을 쓴다. 분리 파서를 만드는 것은 오버엔지니어링으로 판단.

## 6. 응답 수치 정합성 — 3중 장치 (이 문서의 핵심)

**장치 ① 후보 카드는 코드가 조립한다.**
후보·비교표의 수치(위험등급, 총보수, 수익률+기준기간, 주요 보유종목, 기준일)는 LLM 응답 텍스트가 아니라 **funds.json에서 코드가 직접 조립한 구조화 카드**로 UI에 표시한다(04의 후보 카드 컴포넌트). LLM은 "왜 이 후보들인지, 무엇이 다른지" 서술만 담당한다. 수치 환각의 경로를 표시 단계에서 원천 차단하는 구조다.

**장치 ② LLM 컨텍스트는 정형 JSON 발췌만.**
응답 생성 시 LLM에게는 해당 후보들의 funds.json 발췌(필요 필드만)를 제공하고, 프롬프트에 "제공된 JSON에 없는 수치는 생성 금지, 없으면 '데이터에 없다'고 답하라"를 명시한다. RAG CONTENT는 컨텍스트에 넣지 않는다(1장 원칙).

**장치 ③ 후처리 수치 대조 (lite).**
postprocess(02 7-2)에서 응답 텍스트 내 `%` 수치를 정규식으로 추출해, 이번 턴에 제공한 JSON 발췌·계산 결과에 존재하는 수치 집합과 대조한다. 불일치 수치가 있으면 해당 문장을 치환하고 trace에 경고를 남긴다(시연 포인트) — 카드·비교표가 있는 턴은 "자세한 수치는 함께 표시된 카드에서 확인하실 수 있어요", 없는 턴(질문·일반 설명)은 "구체적인 수치는 지금 확인된 데이터에 없어 말씀드리지 않을게요"로 치환한다(2026-07-18 개정: 카드 없는 턴에 카드 안내가 들어가는 문제 수정). 치환·재조립 시 원본 줄바꿈 서식은 보존한다. 재생성 루프는 두지 않는다(지연 방지 — 카드가 정답을 이미 보여주므로 치환으로 충분).

**계산 수치**: 비용 원 단위 환산은 `calc_annual_cost`(02 6장)의 반환 문자열을 그대로 응답에 삽입하고 LLM이 재서술하지 않는다.

## 7. 검색용 CONTENT — 1펀드 = 1문서 (고객언어형)

생성 규칙: 01 8-3 고객 언어 사전의 **역방향 적용**. 고객이 칠 검색어("안정적인", "수수료 싼", "엔비디아 들어간")와 의미적으로 매칭될 문장을 정형 필드로부터 생성한다. 정확한 수치는 넣지 않는다(검색 전용이므로 불필요, 오염 방지).

```
[KF04001379] 피델리티 미국 증권 자투자신탁(주식-재간접형) Ce
미국 주식에 투자하는 주식형 펀드. 가격 변동과 손실 가능성이 큰 편(다소높은위험).
연간 수수료 부담이 낮은 편. 엔비디아(NVIDIA), 마이크로소프트, 알파벳(구글) 등
미국 대형 기술주를 주요 종목으로 담고 있음.
이런 검색에 맞는 상품: 미국 빅테크 펀드, 기술주 성장주 펀드, 엔비디아 포함 펀드,
수수료 낮은 미국 펀드.
```

- 표현 근거는 전부 정형 필드다: "수수료 부담이 낮은 편" ← `fee_quartile=1`, "가격 변동이 큰 편" ← risk_score≥4, 종목 한글 병기 ← alias 사전 역주입. 근거 없는 수식어는 생성하지 않는다.
- 생성 방식: **템플릿 함수(코드)** 기본. LLM 생성은 311건 배치 비용·검수 부담 대비 이득이 없어 미채택.
- metadata: `{"product_code": ...}` — 검색 결과에서 코드로 funds.json을 역참조하는 열쇠.

## 8. 임베딩·벡터스토어 (수업 3주차 매핑)

| 항목 | 선택 | 사유 |
|---|---|---|
| 임베딩 모델 | `text-embedding-3-small` (1536차원) | 수업 표준. 인덱싱·질의 동일 모델 원칙 준수 |
| 벡터스토어 | `InMemoryVectorStore` | 311건. 외부 DB(FAISS·Chroma 등)는 규모 대비 불필요 |
| 청킹 | 없음 — 1펀드 = 1문서가 이미 원자 단위 | `RecursiveCharacterTextSplitter` 등 splitter 미사용. "문서 구조에 맞는 청킹" 원칙의 적용 결과가 '청킹 안 함'인 케이스 |
| 리랭킹·MMR | 미채택 | 정형 필터가 먼저 후보군을 좁히므로 2단계 검색 불필요 |
| 질의 | 사용자 원문 + 추출 조건 결합 텍스트 | 별도 쿼리 확장 LLM 호출 없음(지연·비용) |

## 9. 검색 플로우 — 02 `search_funds` 구현 상세

```
1) 정형 필터 (funds.json → pandas)
   - risk_score ≤ max_risk_score  (01 4-2 하드 필터, 예외 없음 — 성향 초과 제외 건수·blocked 플래그는 02 6장)
   - tags / region / fund_type 조건 일치
   - target_stock: alias 패턴으로 stocks_raw contains (has_holdings_info=false는 제외 + 카운트)
2) 랭킹
   - cost_sensitive → fee_pct 오름차순 (정형 정렬, 벡터 미사용)
   - 그 외 → 필터 통과 코드 집합으로 제한한 벡터 유사도 상위 k (metadata filter)
3) 결과 조립: 후보 2~4개를 funds.json 원본 값으로 카드화 + excluded_by_risk/blocked 플래그
   + 벡터 랭킹 턴은 후보별 코사인 유사도 scores(소수 3자리) — trace 검색 근거 표시용 (2026-07-18)
```

정형 조건이 명확하면 벡터 검색을 아예 타지 않는다("엔비디아 + 수수료 싼" → 필터+정렬로 종결). 벡터 검색은 "노후 대비로 안정적인 거" 같은 뉘앙스 질의에서 값을 한다. 이 역할 분담 자체가 발표 논거다: RAG를 쓰되, RAG가 잘하는 곳에만.

## 10. 종목 alias 사전 — `alias.json` (시연 커버 최소셋)

```json
{
  "엔비디아": ["NVIDIA", "엔비디아"],
  "테슬라": ["TESLA", "TSLA", "테슬라"],
  "애플": ["APPLE", "AAPL", "애플"],
  "마이크로소프트": ["MICROSOFT", "MSFT", "마이크로소프트"],
  "알파벳": ["ALPHABET", "GOOGL", "알파벳", "구글"],
  "아마존": ["AMAZON", "AMZN", "아마존"],
  "삼성전자": ["삼성전자"],
  "SK하이닉스": ["SK하이닉스", "하이닉스"],
  "TSMC": ["TAIWAN SEMICONDUCTOR", "TSMC"],
  "텐센트": ["TENCENT", "텐센트"]
}
```

- 키 = 고객 표기(응답·카드 표시용), 값 = stocks_raw contains 매칭 패턴(대소문자 무시).
- 겹침 분석(`match_overlap`)과 종목 검색이 같은 사전을 쓴다. 페르소나 대본(C: 엔비디아·테슬라, B: 애플·MS·알파벳)을 커버하는 10개로 시작하고, 필요 시에만 추가한다.

## 11. 용어 설명 지식 — `terms.json` (RAG 아님, 사전)

데이터셋에 용어 가이드 문서가 없으므로, explain 노드용 소형 사전을 직접 작성한다. 문서량이 적어(약 20항목) 벡터 검색이 불필요하며, 키워드 매칭 사전이 더 정확하다. 매칭 실패 시 LLM 일반 지식으로 답하되 "일반적인 설명이며 상품별로 다를 수 있음"을 붙인다.

각 항목: `{"term", "aliases", "explain"(2~3문장, 고객 언어), "misconception"(교정할 오개념, 없으면 null)}`

초기 수록 목록(시연 대본 커버): 펀드 기본 구조, 예금과의 차이(원금 비보장 ★D 오개념), 위험등급 체계, 총보수·수수료, 적립식/거치식, TDF와 빈티지 숫자(★B), 글라이드패스, 환헤지 H/UH(★B), 월지급식·분배금(★D 오개념: 이자 아님), MMF, 인덱스펀드·파생형, 채권형/주식형/혼합형, S&P500, 나스닥100, 투자성향과 성향 초과, 연금 적격, 기준가격, 재간접형.

## 12. 빌드 검증 — assert 목록 (실패 시 빌드 중단)

01 11-2의 시나리오 지원 근거를 빌드 시마다 재현 검증한다. 데이터가 바뀌어도 시연이 깨지지 않게 하는 안전핀이다.

```
- 총 311건, risk_score 결측 0건
- 엔비디아 alias 매칭 25건 / 그중 risk_score≤5가 21건
- TDF 태그 32건, vintage 2015~2055 범위
- 안정형 풀(≤2)에서 income 태그 6건
- AI·반도체 키워드 상품의 최소 risk_score ≥ 4 (교차 시연 전제)
- 페르소나 보유 상품코드 4종(KF04001379, KF04000529, KF04000398, KF04001560)이 funds.json에 존재
- CONTENT 수 == funds 수, 임베딩 차원 1536
```

## 13. 데이터 파일 구조

```
data/
  raw/펀드정보.xlsx        # 원본 (수정 금지)
  funds.json               # 정형 메타 + 파생 (수치의 단일 출처)
  contents.jsonl           # 검색용 CONTENT (1펀드 1문서)
  embeddings.npy           # CONTENT 임베딩 캐시
```
