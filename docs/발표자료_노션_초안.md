# 🧭 AI 펀드 길잡이

**"이 상품을 사세요"가 아니라, "말씀하신 조건으로 찾으면 이런 후보가 있고, 차이는 이렇습니다."**

고객의 막연한 언어를 상품 데이터로 번역해, 스스로 발견하게 돕는 개인화 AI 탐색 에이전트

`LangGraph` `RAG` `gpt-4o-mini` `Streamlit` · 실데이터 311개 펀드

## 1. 문제 — 정보는 많은데, 탐색이 안 된다

기존 모바일 펀드 화면은 정보와 필터가 풍부하지만 **고객 주도 탐색형 UI**다.

- 처음 접하는 고객: 무엇을 기준으로 봐야 할지 모른다
- 용어가 어려운 고객: 학습과 상품 탐색이 분리되어 있다
- 조건이 있는 고객도: "엔비디아 많이 들어간 거"를 직접 필터 조합으로 번역해야 한다
- 이해 수준이 달라도 같은 목록, 같은 설명

> **문제의 본질은 상품 정보의 부재가 아니라, 고객의 언어·목적·상황과 상품 데이터 사이의 '탐색 공백'이다.**

## 2. 솔루션 컨셉 — 검색기가 아니라 '발견의 대화'

에이전트는 ① **듣고**(목적·시점 확인) ② **되짚어주고**(탐색 가능한 조건으로 번역해 확인) ③ **발견하게 한다**(복수 후보 + 근거 + 차이 + 선택권).

### 설계 3원칙
1. **탐색이지, 추천이 아니다** — "적합합니다/추천드립니다" 금지. 항상 후보 2~4개, 선정 근거, 차이, 선택권
2. **개인화는 판단이 아니라 맥락에** — 질문 줄이기, 눈높이 조정, 비교 기준선. 개인화로 후보를 숨기지 않음
3. **투자성향 ≠ 이번 자금의 조건** — 성향은 가입 가능 위험등급의 상한(규제 안전선), 이번 돈의 기간·손실감내는 대화로 확인하는 탐색 조건

## 4. 무엇을 할 수 있나 — 행동 4종

| 행동 | 트리거 예시 | 에이전트가 하는 일 |
|---|---|---|
| 🔍 검색 | "엔비디아 많이 들어가고 비용 낮은 거" | 조건 번역 → 하드필터 → 정형/벡터 랭킹 → 후보 카드(근거·기준일) |
| 💬 설명 | "TDF 숫자가 뭐예요?" | 용어 사전 20항목 + 오개념 교정 + 탐색으로 연결 |
| ❓ 질문 | "돈 좀 굴려볼까 하는데요" | 부족한 맥락 1개씩(최대 2턴), 답변 선택지 칩 |
| ⚖️ 비교 | "1번과 3번 비교" / "얼마나 겹쳐?" | 번호로 후보 특정, 정형 비교표, 보유종목 겹침(종목명 기준) |

지원 범위: 복수 종목 AND("테슬라랑 엔비디아 둘 다"), 오타·구어체("엔비댜 잇어?"), 조건 변경("미국 말고 중국으로"), 모호 질의("반토막은 무서워요"), 비용 원 단위 환산("월 20만원이면 첫해 약 1만 1,700원")

## 5. 아키텍처 — 자율성을 제한한 Agent 설계

[📸 LangGraph 구조 다이어그램 — Gemini 생성본]

> **단순 API 호출도 아니고, 통제 불가능한 자율 에이전트도 아니다.**
> ReAct 자율 루프 대신 **명시적 분기 그래프**: 라우터가 행동 4종 중 하나를 고르고, 각 행동 노드가 정해진 도구를 코드로 호출한다.

- **턴당 LLM 호출 정확히 2회** (라우터 + 응답 생성) — 빠르고, 싸고, 디버깅 가능
- LLM의 역할 2가지로 제한: ① 행동 분류+조건 추출(structured output) ② 문장 생성
- 검색 필터·비용 계산·겹침 매칭·안내 문구는 **전부 코드가 보장**
- 라우터 위에 **코드 오버라이드 4규칙** (연속 질문 제한, 비교 폴백, 겹침 강제, 오추출 제거) — LLM이 흔들려도 행동이 예측 가능

### 기술 선택 (Q&A 방어용 — 채택하지 않은 것과 그 이유)

| 패턴 | 판단 | 이유 |
|---|---|---|
| ReAct (`create_agent`+`bind_tools`) | ✘ | 금융 응답은 예측 가능해야. 행동이 4종으로 유한 → 명시 분기가 적합 |
| 멀티에이전트 (Supervisor) | ✘ | 단일 도메인·단일 여정 — 분리로 얻는 게 없음 |
| Store 장기기억 | ✘ | 세션 내 개인화만 (개인정보 최소화). v2 확장 여지 |
| HITL interrupt | ✘ | 가입·실행이 비범위라 승인 게이트 대상 없음 |
| 외부 벡터DB·리랭킹 | ✘ | 311건 규모 — InMemoryVectorStore로 충분 |
| MCP 서버 분리 | 스트레치 | search_funds를 @mcp.tool()로 감싸는 확장 구조로 언급 |

## 6. RAG 설계 — 검색은 고객 언어로, 수치는 정형 조회로

```
펀드정보.xlsx (311건, 원본)
 ├→ contents.jsonl  고객언어형 CONTENT → 임베딩 → 벡터 검색  [검색 전용]
 └→ funds.json      정형 메타+파생 필드 → 필터·정렬·계산     [수치의 단일 출처]
```

- **물리적 분리**: 검색용 문장은 LLM 컨텍스트에 넣지 않는다 → "검색이 잘 되게 쓴 문장"이 답변 수치를 오염시킬 경로 자체가 없음
- **역할 분담이 데이터로 증명됨**:
  - 명확한 조건("엔비디아+비용") → 벡터를 아예 안 탐. 정형 필터+정렬 100%
  - 모호한 질의("노후 대비", "반토막 무서워") → 벡터 P@5 **5/5**
  - "매달 분배금 나오는 상품": 벡터 단독 2/5 → **태그 필터+벡터 결합 4/4**
- 임베딩 품질: TDF 2030의 최근접 이웃 = TDF 2035·2040·2045 (유사도 0.99~0.96)

## 7. 안전장치 — "LLM이 실수해도 코드가 막는다"

### 3중 수치 정합성
① 후보 카드는 funds.json에서 **코드가 조립** (LLM은 서술만) ② LLM 컨텍스트는 정형 JSON 발췌만 ③ 응답 속 %를 제공 수치 집합과 대조 — 불일치 문장은 "카드에서 확인하세요"로 교체

### 투자성향 노출 규칙 (하드 필터)
성향 5단계 → 위험등급 노출 상한 (안정형2 / 안정추구3 / 중립4 / 적극5 / 공격6). 초과 상품은 후보로 노출하지 않되, **차단 사실·사유·대안을 안내** — "차단하되, 차단했다는 사실은 숨기지 않는다"

### 출력 가드 (postprocess, LLM 미사용)
금칙어 3범주 치환(추천·권유형 / 원금보장·오인형 / 최상급형) → 차단 안내 상수 삽입(누락률 0% 구조) → 고지 문구 자동 삽입

### 판매규제 취지의 설계 반영

| 취지 | 구현 장치 |
|---|---|
| 적합성 원칙 | 성향 기반 노출 차단 + 차단 안내 |
| 설명의무 | 후보마다 위험등급·비용·기준일 동시 표기, 원금손실 고지 |
| 부당권유 금지 | 금칙어 코드 치환, 판단 위임 재프레이밍("뭘 사면 돼요?" → 기준 제시 + 선택권 반환) |
| 허위·과장 금지 | 수익보장 표현 차단, 수익률에 기준기간 필수 |

### 개발 중 실제로 막은 사례 (가드레일이 일한 증거)
- LLM이 고객 보유종목을 "펀드가 보유 중"이라고 왜곡 서술 → 컨텍스트 격리로 차단
- "반토막 무서워요"에서 말한 적 없는 "월지급 상품" 조건을 추정 생성 → 코드 검증 규칙으로 제거
- 근거 없는 수익률 수치 생성 → 수치 대조가 문장 교체
- 차단 후 대안 버튼이 다시 차단되는 루프 → 차단 유발 조건 자동 해제

## 8. 개인화 — 은은하게, 3단 체감

**첫 화면에서 보이고** (4인 4색 오프닝·칩) → **질문이 줄어드는 것으로 느껴지고** (세션 조건 누적, 재질문 없음) → **비교에서 증명된다** (보유종목 겹침 분석)

- 페르소나는 테스트 데이터일 뿐 — 런타임이 나이·ID로 분기하지 않음. 발화의 구체성과 세션 맥락으로 행동 선택
- 금지: 행동 단정("다시 오셨네요"), 개인 데이터 복창, 개인화를 이유로 한 후보 숨김

## 9. 품질 검증 — 스테이지 게이트로 만들었다

14단계 게이트: 각 단계마다 산출물 → 자동 검증 → 눈 확인 → 통과 후 커밋. **문서가 계약** — 구현이 어긋나면 문서를 먼저 고침.

| 검증 자산 | 결과 |
|---|---|
| 데이터 빌드 assert 10종 | 전부 PASS (재빌드마다 자동) |
| 라우터 행동 분류 12케이스 | 12/12 × 3회 재현 |
| 대본 수용 pytest (4 페르소나+교차+재프레이밍) | 6/6 × 2회 |
| 강건성: 일반 질의 14종 + 멀티턴 3세트 | 불변식 위반 0 (하드필터·금칙·차단·고지) |
| 검색 품질 평가 | 뉘앙스 질의 P@5 100%, 복합조건 6케이스 |

## 10. 한계와 확장

**정직한 한계 (데이터 기반)**: 종목별 편입 비중·글라이드패스·분배 내역은 공개 데이터에 없음 → 계산하지 않고 한계를 고지 (겹침은 종목명까지, TDF는 빈티지·비용·위험등급 중심)

**비범위(설계 결정)**: 가입·매수 실행 / 수익 예측 / 추천·선택 대행 / 리밸런싱 조언 / 크로스 세션 기억

**확장 방향**: MCP 도구 서버 분리 → 타 채널 재사용 / 고객 동의 기반 탐색 노트 저장("이어보기") / Langfuse 관측 / 실계좌 연동

## 부록 A. 예상 Q&A

- **Q. 왜 ReAct를 안 썼나?** 금융 응답의 예측 가능성. 행동이 4종으로 유한하면 명시 분기가 자율 루프보다 안전하고 빠르다 (턴당 LLM 2회 고정).
- **Q. 추천이랑 뭐가 다른가?** 단수 정답 대신 복수 후보+근거+차이+선택권. 판단형 표현은 코드가 치환. "뭘 사면 돼요?"에는 기준을 정리해주고 선택권을 돌려준다.
- **Q. 환각은 어떻게 막나?** 수치는 LLM이 아니라 funds.json에서 코드가 카드로 조립. 응답 속 %는 제공 집합과 대조해 불일치 문장 교체.
- **Q. 성향 차단이 과하지 않나?** 가입 가능 상한이라는 제도 취지의 반영. 차단을 숨기지 않고 사유·대안(재진단 안내 포함)을 제공.
- **Q. 311건이면 RAG가 필요한가?** 정형 필터가 주력이고, 벡터는 "노후 대비 안정적인 거" 같은 고객 언어 뉘앙스 전용. 역할 분담을 측정으로 확인했다(§6).
